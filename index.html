About

Everything Claude Code is missing, in one plugin.

Claude Code is powerful — but it has no memory between sessions, no undo beyond git, no test feedback, no awareness of its own context window, and no way to detect when it's stuck in a loop. Claude Loop adds all of that.

What Claude Code lacks

  • No persistent memory — every session starts from zero
  • No file-level rollback — bad edits need manual git recovery
  • No automatic test runs after edits
  • No loop detection when Claude repeats a failed approach
  • No context-pressure awareness before hitting the limit
  • No visual regression checks on frontend changes

What Claude Loop adds

  • Persistent structured memory across sessions
  • Automatic file snapshots with one-click rollback
  • Test feedback loops — related tests run after each edit
  • Loop detection that warns Claude when it's stuck
  • Context-pressure monitoring before the window fills up
  • Session compression — decisions & patterns extracted

How it runs

  • Local background worker with hooks into Claude Code
  • SQLite database at ~/.claude-code-loop
  • No cloud, no telemetry, no API keys
  • Install once, forget about it
  • Live dashboard to inspect, diff, test, and roll back

Why this matters

Without memory, Claude repeats mistakes. Without rollback, bad edits require manual recovery. Without test feedback, broken code ships silently. Claude Loop closes these gaps so you spend less time supervising and more time building.

CLI

Install and use from your terminal.

One command installs everything. Claude Loop runs in the background and hooks into every Claude Code session on your machine automatically.

bash — claude-code-loop
$ npx claude-code-loop init   # install hooks + start worker
$ npx claude-code-loop start  # start the background worker
$ npx claude-code-loop status # worker health & sessions
$ npx claude-code-loop uninstall # remove hooks, stop worker
$

How memory works

Claude Loop does not replay full raw conversations. It stores compressed, structured knowledge — decisions, failures, patterns, project setup. Each session gets a compact context pack (~600 tokens) of the most relevant memories. That's how it saves time and reduces repeated work without inflating your context window.

Loop Library

What is a loop, and why is it powerful?

A normal prompt fires once and hopes for the best. A loop turns that single shot into a cycle — try something, check the result, decide the next move, and stop the moment the goal is genuinely met. Pick a tab below and copy a ready-to-run loop straight into Claude Code.

A one-shot prompt

"Speed up the site."

Fires once. No yardstick, no feedback, no signal for whether it worked or when to stop.

The same intent, as a loop

"Profile the pages, pick the slowest, ship one targeted fix, then re-measure. Keep the change only when the numbers actually move — then go again. Stop once every page clears its budget, or a full pass no longer earns its keep."

Performance

Speed Sweep

Use when the app feels sluggish and you want measured, page-by-page wins instead of one risky rewrite.

Profile every page and find the slowest one. Ship a single targeted fix, then measure again. Keep the change only when the numbers genuinely improve, then move to the next page. Stop once every page clears its performance budget — or once a full pass no longer makes a meaningful difference.
Verify: measured load times fall below budget and stay there.

Reliability

Flake Killer

Use when your CI is red for no real reason and intermittent tests are eroding trust in the suite.

Run the suite and isolate the test that fails most often without a genuine code cause. Apply one focused fix, then run that test ten times back to back. Keep the fix only if all ten pass. Move to the next flaky test and repeat until the suite is consistently green or no clear candidate remains.
Verify: ten consecutive green runs per repaired test.

Maintenance

Docs Drift Fixer

Use when the README and the code have quietly drifted apart and examples no longer run.

Scan for a single place where the documentation no longer matches the code. Bring it back in sync and run the example to prove it works. Repeat one mismatch at a time until docs and code agree. Pause and check with me before removing any section outright.
Verify: every example runs; no stale references left.

Quality

Coverage Climber

Use when critical paths are under-tested and you want coverage that reflects real behavior, not vanity numbers.

Locate the most under-tested critical module. Add one test that exercises real behavior — not a placeholder — and re-run coverage. Keep it only if coverage rises and the suite stays green. Repeat until you reach the target or new tests stop earning their place.
Verify: coverage increases; all tests pass.

Production

Error Triage

Use when production is noisy and you want to quiet the loudest errors first, safely.

Pull the latest production errors and sort them by frequency. Reproduce the top one, ship a focused fix, and watch the rate respond. Work down the list and repeat until the loudest errors go quiet. Stop and ask me first before any change that touches data, billing, or auth.
Verify: error frequency falls in monitoring.

Review

Diff Polisher

Use when a change is nearly ready and you want one more careful pass before it ships.

Read the current diff and surface one concrete correctness or clarity problem. Fix it, re-run the affected tests, and keep the change only if they stay green. Repeat until nothing material is left, then write a short note on what changed and why.
Verify: tests pass; every fix is justified.

Eight ways to work with loops

From discovery to publication.

These ideas build on the open-source Loopy / Loop Library project. The throughline: every solid loop names its target, defines how it will know it succeeded, decides what to do with each result, and sets the point where it stops or hands back to you.

Discover

Comb a repo or your past sessions for work you keep redoing, and shape the strongest candidate into a bounded, repeatable loop.

Find

Browse a catalog of proven loops and surface the few that match the outcome you're actually after.

Loop Doctor

Stress-test a loop for soft checks, risky steps, or fuzzy stop conditions — then patch only what genuinely matters.

Adapt

Reshape an existing loop around your real stack, constraints, cadence, and bar for "done".

Craft

Answer a few quick questions about goal, success, scope, and limits, and walk away with a loop tailored to them.

Run

Drive a loop through capped passes, apply its check after each one, and hand back a receipt backed by evidence.

Debrief

Read the receipts from finished runs and suggest the single smallest change worth making to the loop.

Publish

Vet quality, check for overlap, preview the exact result, and share a loop only once you've signed off.

Loops are bounded by design

A loop is never a licence to run forever. The good ones carry a concrete check, a hard stopping point, and a moment that returns control to a human whenever judgement or sign-off is needed. Claude Loop's memory, rollback, and test feedback are precisely what make running these loops safe.
